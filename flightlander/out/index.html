The Calculus of Images

The Arithmetic of Images is an interesting section in How To Design Programs touching on a something that comes up a lot in games or any other kind of graphical programming.

One of Eddi Woo's excellent youtube lessons What is the basic building block of all mathematics? (A surprising answer!) explains that sets and what operations hold between elements of a given set form the foundation of the subject.

This used to be called calculus, which has become synonymous with "the calculus of infinitesimals", thereby robbing language of an important concept which is possibly why the HTDP authors used arithmetic for the same concept.

I prefer calculus of images since we can actually also use images to do differentiation and integration.

What is the type of an image?

JavaScript's Image Object is the natural choice, but it took quite a bit of frustration to figure out how to get functions to return this object instead of drawing on the canvas. It involved trip down the rabbit hole of ImageData and other extraneous stuff, but I eventually found what I wanted was toDataURL which stores the object as a png bitmap (for some reason, keeping drawings in vector format would require third party libraries).

Images are not created in the actual canvas, but in a hidden doodling area:

const shadowCanvas = document.createElement("canvas");
const shadowCtx = shadowCanvas.getContext("2d");

An image creation function follows these steps:

  1. The bounding box — maximum width and height of the given shape — is calcuated. On gotcha is for stroked shapes, the width of the line needs to be added to these.
  2. The shadow canvas is sized to this box
  3. shadowCtx.clearRect(0, 0, width, height);
  4. The shape gets drawn on the shadowCtx using some of the CanvasRenderingContext2D's many, many methods
  5. image.src = shadowCanvas.toDataURL(); return image;

Drawing traditional maths graphs

The first gotcha in trying to use Cartesian co-ordinates is the center, ie x=0 and y=0, is the top left corner. Is is easily moved to wherever we want the center to be with

  ctx.translate(x0, y0);

The second gotcha is the computer graphics convention is for down to be positive and up negative, so the sign of y needs to be flipped (ie -y).
